Modelos de base de datos - Guia visual

Esta pagina convierte la explicacion textual del modelo de datos en una guia para estudiar de forma rapida y visual. El objetivo es entender que guarda la app, como se relacionan User, CampusUser, Coalition, snapshots, cursores y datos propios como amigos o preferencias.

Mapa visual inicial

Piensa en el sistema como un tablero con siete bloques grandes: identidad interna, perfil sincronizado, estado actual de coaliciones, historico, sync incremental, metadatos operativos y datos propios de producto.

👤

User = identidad interna Django

Guarda la identidad base de autenticacion, sesion, permisos y cookies. No es el perfil rico de 42.

🛰️

CampusUser = perfil sincronizado desde 42

Es el modelo central del dominio: login, intra_id, nivel, scores, proyectos, correcciones y estado de coalicion.

🏛️

Coalition = estado local actual

Representa cada coalicion con su score total actual, slug, imagen, color y lider conocido.

📸

Snapshots = historico

Fotos diarias para comparar evolucion de scores de coaliciones y usuarios sin consultar la API de 42 cada vez.

🧭

Cursors = sync incremental

Recuerdan el ultimo score event procesado por coalicion para no volver a contar proyectos o evaluaciones.

🧾

SyncMetadata = ultimo sync

Tabla pequeña de control: guarda cuando se actualizo una parte del sistema y alimenta observabilidad.

🤝

FriendsList / UserPreferences = datos propios de la app

No vienen de 42. Son capas de producto: amistades, solicitudes, avatar custom, items por pagina, tema y privacidad.

Diagrama principal de relaciones

Aqui se ve la arquitectura relacional local: User cuelga de auth, CampusUser modela el dominio, Coalition concentra estado actual y de ahi salen snapshots y cursores. SyncMetadata queda separado porque es metadato operativo, no dominio de usuario.

Relacion principal de dominio Cursor incremental por coalicion Metadato operativo separado

Lectura rapida: User es la base interna de auth. CampusUser es el perfil rico de 42. Coalition guarda el estado actual. De esos dos salen los snapshots historicos. Los cursors son OneToOne por coalicion para sync incremental. SyncMetadata vive al margen porque describe el estado del proceso de sync, no una entidad de producto.

Diagrama de flujo de datos

Este pipeline es clave en evaluacion: el frontend no llama directamente a 42. La app se apoya en una base local sincronizada y luego sirve consultas rapidas desde Django y PostgreSQL.

Idea fuerza: la app usa la API de 42 para alimentar la base local, no para responder cada pantalla en tiempo real. Eso hace posible tener ranking, snapshots historicos, pagina de coaliciones y endpoints estables incluso si 42 es lenta o tiene rate limit.

User

Usuario interno de Django para autenticacion, sesion y permisos.

Builtin Django
  • Archivo: modelo builtin de django.contrib.auth.models.User
  • Tabla aproximada: auth_user
  • Que representa: identidad interna para login, sesion y permisos.
  • Problema que resuelve: separar auth interna de los datos academicos traidos desde 42.

Campos principales:

id username email password is_active

Relaciones:

  • OneToOne con CampusUser mediante django_user.
  • OneToOne con FriendsList.
  • OneToOne con UserPreferences.
  • Quien lo usa: autenticacion OAuth 42, vistas de profile, friends y preferencias.
  • Riesgo: confundirlo con el modelo principal del dominio. No es donde viven scores ni ranking.
  • Frase para evaluacion: "User resuelve auth y sesion; el perfil academico vive en CampusUser".
FUNCIÓN enlazar_user_con_campus_user():

    crear o localizar User interno
    buscar CampusUser por intra_id

    SI existe:
        campus_user.django_user = user
        guardar relacion

    devolver sesion autenticada
from django.contrib.auth.models import User

django_user = models.OneToOneField(
    User,
    on_delete=models.SET_NULL,
    related_name="campus_user_profile",
    null=True,
    blank=True,
)

CampusUser

Modelo central del dominio: identidad y metricas sincronizadas desde 42.

sync_campususer
  • Archivo: backend/sync/models.py
  • Tabla aproximada: sync_campususer
  • Que representa: el usuario de 42 ya normalizado para consultas locales.
  • Problema que resuelve: guardar en PostgreSQL el perfil rico y las metricas sin depender de la API externa en tiempo real.

Campos importantes:

intra_id user_id login display_name level wallet correction_points projects_delivered_total evaluations_done_total coalition_user_score general_rank is_active

Relaciones:

  • OneToOne opcional con User mediante django_user.
  • ForeignKey inversa desde CampusUserScoreSnapshot.
  • Relacion denormalizada con coalicion via coalition_id, coalition_name, coalition_slug.
  • Quien lo usa: sync, auth callback, coalitions, users, leaderboard y frontend.
  • Riesgo: es un modelo grande y denormalizado; si el sync esta viejo, todas las pantallas de negocio se degradan.
  • Ejemplo de dato: intra_id=217446, login="rodde-fr", coalition_user_score=241000.
  • Frase para evaluacion: "CampusUser es el modelo central del dominio, no User".
FUNCIÓN sync_campus_user(payload_42):

    buscar CampusUser por intra_id
    actualizar identidad, nivel y metricas
    actualizar datos de coalicion actual
    enlazar con User si existe login asociado
    guardar registro local
intra_id = models.PositiveBigIntegerField(unique=True)
login = models.CharField(max_length=255, blank=True)
level = models.DecimalField(max_digits=8, decimal_places=2, default=0)
coalition_user_score = models.IntegerField(default=0)
evaluations_done_total = models.PositiveIntegerField(default=0)

Coalition

Estado actual de cada coalicion en la base local.

sync_coalition
  • Archivo: backend/sync/models.py
  • Tabla aproximada: sync_coalition
  • Que representa: cada coalicion conocida por el sistema.
  • Problema que resuelve: centralizar score total, apariencia visual y metadatos de ranking para frontend y API.
coalition_id name slug image_url cover_url color total_score leader_user_id updated_at
  • Uno a muchos con CoalitionScoreSnapshot.
  • OneToOne con CoalitionProjectCursor.
  • OneToOne con CoalitionEvaluationCursor.
  • Quien lo usa: sync, coalitions services, leaderboard, public_api de coaliciones.
  • Riesgo: solo guarda el estado actual; para tendencia historica necesitas snapshots.

    actualizar nombre, slug, color e imagenes
    actualizar total_score y leader_user_id
    guardar updated_at automatico

slug = models.CharField(max_length=255, blank=True)

CoalitionScoreSnapshot

Foto diaria del score total de una coalicion.

sync_coalitionscoresnapshot
  • Que representa: historico diario de score y campus rank por coalicion.
  • Problema que resuelve: calcular tendencias 24h/7d/30d sin recalcular desde la API de 42.
  • Tabla aproximada: sync_coalitionscoresnapshot
  • Fuente: creada desde el modulo de sync.
coalition snapshot_date total_score campus_rank captured_at
  • ForeignKey a Coalition con related_name="score_snapshots".
  • Restriccion: un snapshot por coalicion y dia (unique_together).
  • Indice por (coalition, snapshot_date).
  • Quien lo usa: sync.services y servicios de coaliciones para tendencias.
  • Riesgo: si faltan snapshots, las graficas historicas son pobres o planas.
  • Frase para evaluacion: "Un snapshot no es el estado actual; es una foto diaria para comparar evoluciones".
FUNCIÓN snapshot_coalition_hoy(coalition):

    SI ya existe snapshot para hoy:
        no duplicar

    SI no existe:
        guardar total_score actual
        guardar campus_rank actual
        marcar captured_at
coalition = models.ForeignKey(Coalition, on_delete=models.CASCADE,
    related_name="score_snapshots")
snapshot_date = models.DateField()
total_score = models.IntegerField(default=0)
campus_rank = models.PositiveIntegerField(null=True, blank=True)

CampusUserScoreSnapshot

Foto diaria del score de un CampusUser.

sync_campususerscoresnapshot
  • Que representa: historico diario del score y ranking local de un usuario.
  • Problema que resuelve: comparar evolucion individual sin depender de datos en vivo.
  • Tabla aproximada: sync_campususerscoresnapshot
  • Contexto: es el equivalente individual del snapshot de coalicion.
campus_user snapshot_date coalition_user_score coalition_user_rank campus_user_rank captured_at
  • ForeignKey a CampusUser.
  • Restriccion: un snapshot por usuario y dia.
  • Indice por (campus_user, snapshot_date).
  • Quien lo usa: sync y comparativas de ranking / perfil.
  • Riesgo: puede crecer mucho si la base tiene miles de usuarios y muchas fechas.
  • Frase para evaluacion: "CampusUserScoreSnapshot permite historico individual sin tocar la API 42".
FUNCIÓN snapshot_usuario_hoy(campus_user):

    leer score actual y ranks actuales
    SI hoy no existe registro:
        insertar snapshot

    devolver historico disponible
campus_user = models.ForeignKey(CampusUser, on_delete=models.CASCADE,
    related_name="score_snapshots")
snapshot_date = models.DateField()
coalition_user_score = models.IntegerField(default=0)
campus_user_rank = models.PositiveIntegerField(null=True, blank=True)

CoalitionProjectCursor

Cursor incremental para eventos de proyectos por coalicion.

sync_coalitionprojectcursor
  • Que representa: ultimo score event de proyectos ya procesado para una coalicion.
  • Problema que resuelve: evita volver a sumar eventos antiguos en sync incremental.
  • Tabla aproximada: sync_coalitionprojectcursor
  • Patron: un cursor por coalicion.
coalition last_score_id last_score_created_at last_synced_at
  • OneToOne con Coalition.
  • Usado por backend/sync/projects.py.
  • Riesgo: si el cursor se posiciona mal, puedes duplicar o saltarte proyectos.
  • Quien lo usa: sync incremental de delivered projects.
  • Frase para evaluacion: "El cursor convierte un recuento caro en un sync incremental".
FUNCIÓN sync_incremental_proyectos(coalition):

    leer cursor actual
    pedir scores nuevos a 42

    SI no hay cursor:
        bootstrapear y salir

    aplicar solo eventos posteriores al last_score_id
    actualizar cursor con el score mas nuevo
coalition = models.OneToOneField(
    Coalition,
    on_delete=models.CASCADE,
    related_name="project_cursor",
)
last_score_id = models.PositiveBigIntegerField(null=True, blank=True)

CoalitionEvaluationCursor

Cursor incremental para correcciones y evaluaciones.

sync_coalitionevaluationcursor
  • Que representa: ultimo score event de evaluaciones procesado para cada coalicion.
  • Problema que resuelve: mantener contadores de correcciones sin recontar todo el historial.
  • Tabla aproximada: sync_coalitionevaluationcursor
  • Patron: mismo esquema que ProjectCursor pero para otro flujo.
coalition last_score_id last_score_created_at last_synced_at
  • OneToOne con Coalition.
  • Usado por backend/sync/evaluations.py.
  • Riesgo: si no conoces el punto de corte del snapshot, el incremental no es fiable.
  • Quien lo usa: sync incremental de evaluaciones/correcciones.
  • Frase para evaluacion: "Hay un cursor por coalicion, no un cursor global".
FUNCIÓN sync_incremental_evaluaciones(coalition):

    leer evaluation_cursor
    traer eventos nuevos desde 42
    filtrar solo reason de evaluacion
    sumar en CampusUser usando coalitions_user_id
    mover cursor al evento mas nuevo
coalition = models.OneToOneField(
    Coalition,
    on_delete=models.CASCADE,
    related_name="evaluation_cursor",
)
last_score_created_at = models.DateTimeField(null=True, blank=True)

SyncMetadata

Tabla minima para saber cuando se actualizo el sync.

sync_syncmetadata
  • Que representa: metadatos del proceso de sync, no una entidad de negocio.
  • Problema que resuelve: poder decir "ultimo sync" en status page y servicios.
  • Tabla aproximada: sync_syncmetadata
  • Forma: pares key / last_time_update.
key last_time_update
  • Sin ForeignKey a dominio.
  • Consumido por /api/status/ y servicios que necesitan saber frescura del sync.
  • Riesgo: si la tabla no existe o no se actualiza, status y observabilidad fallan o mienten.
  • Quien lo usa: config.views y sync.services.
  • Frase para evaluacion: "SyncMetadata no modela usuarios; modela salud operativa del sync".
FUNCIÓN actualizar_sync_metadata(clave):

    buscar registro por key
    SI no existe:
        crearlo

    guardar last_time_update = ahora
    devolver timestamp
class SyncMetadata(models.Model):
    key = models.CharField(max_length=64, unique=True)
    last_time_update = models.DateTimeField(
        null=True,
        blank=True
    )

FriendsList

Capa social local: amigos y solicitudes.

users_friendslist
  • Archivo: backend/users/models.py
  • Tabla aproximada: users_friendslist + tablas intermedias M2M.
  • Que representa: relaciones sociales propias del producto.
  • Problema que resuelve: mantener amistades y solicitudes fuera de la API de 42.
owner friends friends_requests_received friends_requests_sent
  • OneToOne con User via owner.

FUNCIÓN enviar_solicitud_amistad(owner, target):

    añadir owner a friends_requests_received
    guardar ambos lados

friends = models.ManyToManyField("self", symmetrical=True, blank=True)
friends_requests_received = models.ManyToManyField(

Glosario visual

Un repaso corto de vocabulario que suele salir en defensa tecnica.

Modelo

Clase Django que representa una tabla o entidad persistida.

Tabla

Representacion SQL real en PostgreSQL donde se almacenan las filas.

ForeignKey

Relacion muchos-a-uno entre modelos.

OneToOne

Relacion uno-a-uno: un registro enlaza con uno solo del otro lado.

ManyToMany

Relacion N a N apoyada por una tabla intermedia.

Snapshot

Foto historica guardada en una fecha concreta.

Cursor

Punto de reanudacion para sync incremental.

Migracion

Instruccion versionada para crear o cambiar tablas.

SyncMetadata

Metadatos operativos del proceso de sincronizacion.
