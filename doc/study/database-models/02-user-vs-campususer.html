02. User vs CampusUser

Separación entre autenticación interna y dominio sincronizado desde 42, con foco en la relación OneToOne y la sintaxis asociada.

🧩 Django ORM 🐘 PostgreSQL 🌐 API 42 🔄 Sync local 📚 Referencias al repo

Diferencia conceptual

User resuelve autenticación y sesión. CampusUser resuelve el dominio académico sincronizado desde 42. Esa separación evita mezclar auth interna con métricas de negocio.

User

  • Tabla: auth_user
  • Rol: login, sesión y permisos
  • No guarda: score, rank, coalición ni métricas

CampusUser

  • Tabla: sync_campususer
  • Rol: identidad 42 + métricas + estado local
  • Sí guarda: intra_id, level, login, score, rank y coalición

Referencia: doc/database-models-explained.md, sección Diferencia entre piezas principales + 4.1 + 4.5, línea aproximada 24

Relación OneToOne

El enlace real vive en CampusUser.django_user. Es opcional, usa SET_NULL y permite conservar el perfil de campus aunque desaparezca el User interno.

from django.contrib.auth.models import User

django_user = models.OneToOneField(
    User,
    on_delete=models.SET_NULL,
    related_name="campus_user_profile",
    null=True,
    blank=True,
)
Cómo leerlo

Un CampusUser puede enlazar con un único User. Si borras el User, el campo queda a NULL; no se borra el CampusUser.

FUNCIÓN enlazar_user_con_campus_user(user, intra_id):

    SI existe campus_user:
        campus_user.django_user = user

Sintaxis ORM destacada

Esta página adelanta la sintaxis más crítica porque aparece justo en la relación User ↔ CampusUser.

OneToOneField

Un registro apunta a uno solo del otro lado. Aquí evita que un mismo User tenga múltiples CampusUser enlazados.

Referencia: doc/database-models-explained.md, sección 5. OneToOneField, línea aproximada 947

SET_NULL

Si se borra el padre, el enlace queda a NULL. Protege el histórico del CampusUser.

Referencia: doc/database-models-explained.md, sección 5. on_delete, línea aproximada 947

null=True y blank=True

null afecta la base; blank la validación. Juntos hacen el enlace opcional.

Referencia: doc/database-models-explained.md, sección 5. null=True / blank=True, línea aproximada 947

unique=True

Garantiza unicidad real en BD. En intra_id evita duplicar perfiles del mismo alumno 42.

Riesgos y errores comunes

Lo que más se confunde en evaluación.

  • Error oral típico: decir que User es el modelo principal del dominio. No lo es; lo es CampusUser.
  • Riesgo técnico: si el sync de CampusUser está viejo, leaderboard, perfiles y coaliciones pierden frescura.
  • Error de sintaxis: confundir SET_NULL con CASCADE.
  • Error de diseño: olvidar que el enlace a User es opcional y no siempre existe.
