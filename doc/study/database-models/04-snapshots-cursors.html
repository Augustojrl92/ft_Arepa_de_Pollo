04. Snapshots y Cursors

Aqui aprendes a no confundir historico con incremental. Ambos conceptos son necesarios, pero resuelven problemas distintos.

Django ORM PostgreSQL API 42 Sync Study mode

Historico vs incremental

Esta pagina separa dos ideas que suelen confundirse: snapshots guardan fotos historicas; cursors guardan el ultimo evento ya procesado.

📸

Snapshots

  • CoalitionScoreSnapshot
  • CampusUserScoreSnapshot
  • sirven para ver evolucion diaria
  • no aceleran incremental por si mismos
🧭

Cursors

  • CoalitionProjectCursor
  • CoalitionEvaluationCursor
  • sirven para no recontar eventos antiguos
  • no sustituyen al historico

Diagrama visual

El estado actual sale de CampusUser y Coalition. Desde ahi bifurcan dos caminos distintos.

Formula para no confundirte: snapshot = foto historica; cursor = marca de progreso del sync.

Los cuatro modelos clave

Dos modelos de historico y dos modelos de incremental.

CoalitionScoreSnapshot

FK a Coalition, un registro por coalicion y fecha. Guarda total_score y campus_rank.

SI hoy no existe snapshot:
    guardar score actual de la coalicion

CampusUserScoreSnapshot

FK a CampusUser, un registro por usuario y fecha. Guarda score y ranks del dia.

SI hoy no existe snapshot:
    guardar score y rank del usuario

CoalitionProjectCursor

OneToOne con Coalition. Recuerda el ultimo score event de proyectos ya procesado.

leer last_score_id
procesar solo eventos posteriores

CoalitionEvaluationCursor

OneToOne con Coalition. Hace lo mismo pero para correcciones/evaluaciones.

leer last_score_id
sumar solo evaluation events nuevos
